# Auto-Selp 기획서 초안

## 1. 제안 요청

### 1.1 프로젝트 소개
Auto-Selp는 이커머스(네이버 쇼핑, 쿠팡 등) 셀러들을 위한 **상품 정보 자동화 및 대량 처리 솔루션**입니다. 판매자가 업로드한 엑셀 파일(상품명 등)을 바탕으로 LLM(OpenAI, GPT-5-Nano)과 네이버 오픈 API를 활용하여 최적화된 상품명, 경쟁력 있는 키워드, 그리고 카테고리 매핑을 자동으로 수행하여 셀러의 업무 효율성을 극대화합니다.

### 1.2 시장 조사
- **이커머스 셀러 증가**: 온라인 쇼핑 시장의 지속적인 성장으로 신규 셀러가 급증하고 있으나, 단순 반복적인 소싱 및 등록 작업에 많은 시간을 할애하고 있습니다.
- **키워드 및 카테고리 최적화의 중요성**: 상위 노출을 위해서는 검색량 대비 경쟁 강도가 낮은 '황금 키워드' 발굴과 정확한 카테고리 매칭이 필수적입니다.
- **기존 툴의 한계**: 수동 검색이나 단건 처리 위주의 서비스가 많아, 대량의 엑셀 데이터를 한 번에 병렬 처리하고 LLM을 통해 지능적으로 데이터를 가공해주는 통합 솔루션에 대한 니즈가 존재합니다.

### 1.3 유사 프로그램 분석
*추가 정보 필요:* 아이템스카우트, 판다랭크, 셀러마스터 등 기존 키워드 분석 툴과의 구체적인 차별점 및 벤치마킹 대상을 설정해야 합니다. (작성 후 유저 확인 필요)

### 1.4 주요 기능
1. **엑셀 대량 처리 및 병렬 작업 (Parallel Processing)**:
   - 엑셀 업로드 시 컬럼 자동 매핑 및 설정 저장 기능.
   - 멀티 스레딩(1~10개)을 통한 대용량 데이터의 빠른 병렬 처리.
2. **지능형 키워드 생성 및 필터링**:
   - 다단계 파이프라인: 시드 수집 -> 네이버 API 기반 데이터 필터링(경쟁률, 검색량) -> 상표권 검증 -> LLM 최종 큐레이션(최대 10개).
3. **상품명 가공 및 카테고리 자동 매핑**:
   - LLM을 활용한 SEO 최적화 상품명 생성.
   - 엑셀 캐싱 및 네이버 API 풀백을 통한 정확하고 빠른 카테고리 매칭.
4. **프롬프트 관리 (Prompt Management)**:
   - LLM에 사용되는 프롬프트의 CRUD 등 커스텀 관리 기능 제공.
5. **실시간 작업 트래킹**:
   - 최근 활동 로그 화면, 각 단계별(상품명, 키워드, 카테고리) 및 행별 실시간 프로세스 진행률(%) 확인 및 작업 취소 기능.

### 1.5 기대 효과
- **시간 단축**: 대량의 상품 정보를 수 분 내에 최적화하여 단순 소싱 업무에 소요되는 리소스 대폭 절감.
- **매출 증대**: 데이터 기반의 경쟁력 있는 롱테일 키워드 발굴과 상표권 리스크 필터링으로 안전하고 효과적인 상품 노출.
- **운영 편의성**: 실시간 작업 트래킹과 커스텀 프롬프트 설정을 통해 사용자 맞춤형 워크플로우 지원.

### 1.6 개발 환경
- **Frontend**: React (Vite 기반), 다크/화이트 모드 지원 UI 구조.
- **Backend**: Python (멀티스레딩/워커 기반의 병렬 처리 구현).
- **AI/API**: OpenAI, 자체/외부 LLM(GPT-5-Nano), 네이버 쇼핑 오픈 API.
- **인프라/DevOps**: Oracle Cloud(Oracle Linux 9), Docker & Docker Compose 컨테이너화, GitHub Actions 기반 CI/CD 파이프라인.

## 2. 업무 분석 및 요건 정의

### 2.1 업무 분석
- 셀러는 상품 소싱 리스트(엑셀)를 준비하여 Auto-Selp 시스템에 업로드합니다.
- 시스템은 엑셀의 헤더 매핑을 사용자 설정에 맞게 적용하고, 입력된 설정(프롬프트, 병렬 처리 수 등)을 기반으로 백그라운드 작업을 생성합니다.
- 백엔드 워커가 LLM과 외부 API를 호출하여 상품명/키워드/카테고리 데이터를 채워 넣습니다.
- 처리 중 결과는 웹소켓이나 폴링으로 프론트엔드에 실시간 공유되며, 완료 후 결과 엑셀 파일을 다운로드합니다.

### 2.2 요구 사항 명세
1. 화면 해상도 및 테마(모드)에 따라 UI 텍스트와 레이아웃이 가시적으로 표현되어야 함 (특히 화이트 모드 및 가로 스크롤 이슈 대응).
2. LLM 호출 시 한글 인코딩 문제 없이 안정적으로 텍스트를 생성해야 하며, 실패 시 재요청(Retry) 로직이 동작해야 함.

*(유스케이스 다이어그램, 프로토타이핑, 전체/단위 업무 흐름도, 단위 업무 정의서는 다이어그램 및 시각 자료 형태로 시스템 구성에 맞춰 구체화 필요)*

## 3. 기획 & 설계

### 3-1. 데이터베이스 설계
- **논리/물리 데이터베이스 모델 (ERD)**: 
  - `User`, `Prompt`, `JobSequence`, `CategoryMappingCache` 중심의 테이블 설계 필요. *(DB 엔진 종류에 대한 정보 확인 후 작성)*

### 3-2. 기획 및 화면 설계
- **정책정의서**: 회원 가입 및 인증 정책, 상표권 블랙리스트 적용 정책, 병렬 처리 리소스 제한 정책 등.
- **메뉴구조도 (사이트맵)**:
  - 데시보드
  - 엑셀 대량 처리 (분리된 페이지)
  - 프롬프트 관리
  - 작업 내역(Job Status & History)
  - 설정 (API 키, 엑셀 컬럼 매핑 저장 등)

### 3-3. 프로그램 설계
- **프로그램 목록 및 상세**: 
  - 엑셀 파서/라이터 컴포넌트, LLM 프롬프트 생성기 컴포넌트, 네이버 API 연동 모듈 등 모듈별 명세.
- **REST API 설계**:
  - `POST /upload`, `GET /jobs/{id}/status`, `POST /jobs/{id}/cancel`, 프롬프트 CRUD에 대한 `/api/prompts` 설계 등.

### 3-4. 테스트 설계
- **단위/통합 테스트 시나리오**:
  - 엑셀 헤더 매핑 유효성 검사, LLM 호출 결과값 한글 깨짐 테스트, 대량 병렬 처리 시 병목 및 동시성 제어 테스트.
